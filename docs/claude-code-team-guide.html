◈ The Core Idea
⚡
What This Workflow Does

Replaces the traditional dev loop with an AI-assisted one. Claude handles scoping, implementation, review, and documentation. The developer's job shifts from writing boilerplate to making decisions inside a well-scoped boundary.

📊
Without vs. With Claude Code
  • PRD → cards: 2–4 hrs → 20 min via /prd-assess
  • Session context setup: 45 min → 5 min with Skills
  • PR pre-review: 30–60 min → 10 min via Review Buddy Skill
  • Test generation: 1–2 hrs → 20 min from acceptance criteria

◈ The 8-Step Pipeline

01
PRD Received
PM → Lead
▼
What to Do
  • Receive PRD from PM / stakeholder
  • Verify it has all 4 required fields
  • If incomplete — return it. Work does not start.
A Valid PRD Must Have
  • Clear objective
  • Scope — IN and OUT explicitly defined
  • At least one verifiable acceptance criterion
  • PRD number (e.g. PRD-042)
What Can Go Wrong
  • Vague scope → leads to card bloat
  • Missing acceptance criteria → untestable output
  • No PRD number → can't link Jira cards
02
Impact Assessment
Lead · Max tier
▼
Command
/prd-assess docs/prd/PRD-042.md
Claude Reads
  • ARCHITECTURE.md
  • The PRD document
  • All active context MDs
IMPACT-ASSESSMENT.md Output
  • PRD type classification
  • Affected modules + owners
  • Context MDs needing update
  • Skills needing rebuild
  • Estimated cards per role
  • Dependency + cross-project risks
Gate
  • Nothing moves forward until document exists
  • Lead reviews before cards are created
  • Missed dependencies here = rework later
03
Context MDs Updated
Lead
▼
For Each Affected Module
  • Update key files, dependencies, gotchas
  • Set last_verified_sprint to current sprint
  • Update ARCHITECTURE.md if structure changed
  • Rebuild Custom Skill for that module
  • Update CONTEXT-HEALTH.md staleness flags
Context MD Structure
  • last_verified_sprint: Sprint-N
  • Key Files — exact files devs may touch
  • Dependencies — internal + external
  • Do Not Touch — needs Lead approval
  • Gotchas — known bugs, quirks
  • Required Patterns — React Query, Zod, etc.
What Can Go Wrong
  • Stale context MD = devs build in wrong direction confidently
  • Missing gotchas = Safari bugs in PRs
  • Skill not rebuilt = dev gets outdated guidance
04
Jira Cards Created
Lead
▼
Command
/gen-cards docs/context/IMPACT-ASSESSMENT.md
Every Card Must Have
  • PRD reference (no card without this)
  • Skill name or context MD path
  • Exact scope: files you MAY touch
  • Explicit do-not-touch list
  • Acceptance criteria (verifiable)
  • Size: XS, S, or M only — no L or XL
3 Log Fields Pre-Set
  • AI-assisted: Y/N
  • Rework required: Y/N

What Can Go Wrong
  • L/XL card = session limit hit mid-implementation
  • Missing scope = dev touches wrong files
  • Missing log fields = no sprint measurement
05
Implementation
Developer · Pro
▼
Session Start Prompt Template

TASK: [exact task from Jira card]
SCOPE: Touch ONLY [files listed on card]
DO NOT: [do-not-touch list from card]
CONTEXT: [Skill name] loaded / [module.context.md] attached

Session Rules
  • One session per card — restart for each
  • Stop when acceptance criteria are met
  • Limit hit = card was too large → flag and split
  • Never paste raw source files into Claude
On Card Close
  • Fill AI-assisted field (Y/N)
  • Fill rework required field (Y/N)
  • Fill session count
  • Move to Done only after log fields filled
06
Review Buddy + QA
Senior Dev → QA
▼
Buddy Checklist (Skill)

  • Test cases from acceptance criteria

  • Browser matrix: Chrome, Firefox, Safari

  • Buddy fail → returns to dev with comments

Claude Code Security Guidelines

Claude Code runs in your terminal with your permissions. It can read any file, execute commands, and access credentials. These rules are not suggestions — bypassing them is a security incident.

10
Security Rules
3
Real-World CVEs
0
Auto-Accepts Allowed
◈ Why This Matters — Real Incidents
⚠️
CVE-2025-55284

API key theft via DNS exfiltration triggered directly through Claude Code. Credentials were leaked to an external server through a malicious session.

⚠️
GlueStack Supply Chain

17 npm packages with 1M+ weekly downloads were compromised. AI-assisted dependency installs accelerated the spread to developer machines.

⚠️
Rules File Backdoor

Invisible Unicode characters in config files used to poison AI coding tool repositories. The attack was undetectable by visual inspection.

🔴
Claude Code is a powerful tool. Misconfigured, it becomes an attack surface with your credentials attached. Treat it like a brilliant but untrusted intern — capable of excellent work but requiring supervision on everything security-related.
◈ The 10 Security Rules
👆
Click any rule to expand full detail, banned patterns, and what to do.
01
No Auto-Accept. No Bypassed Permissions.
CRITICAL
▼
🚫
Banned: --dangerously-skip-permissions · auto-accepting without reading · "yolo mode" · approving commands you don't understand
Required: Read every permission prompt before accepting. If you don't understand what a command does — deny it and ask the Lead. Only allowlist genuinely safe, repetitive commands.
Team .claude/settings.json Template
{
  "permissions": {
    "allow": ["Bash(git status)", "Bash(git diff*)", "Bash(ls*)", "Bash(echo*)"],
    "ask":   ["Bash(npm install*)", "Bash(git add*)", "Bash(git commit*)"],
    "deny":  ["Bash(rm -rf*)", "Bash(curl*)", "Bash(wget*)", "Bash(git push*)", "Bash(chmod 777*)"]
  }
}
ℹ️
Lead provides the team's base .claude/settings.json at sprint start. Developers do not modify the deny list without Lead approval.
02
Claude Must Never See Secrets or Credentials
CRITICAL
▼
Claude Must Never See
  • API keys (any service)
  • Database credentials or connection strings
  • JWT secrets or session signing keys
  • OAuth client secrets
  • SSH private keys
  • .env files containing any of the above
Required Practices
  • Use .env.example only — never .env in worktree
  • Reference by name only: process.env.DATABASE_URL
  • Use secrets manager CLI, not plaintext files
  • Never paste credential values into a session
🔴
If Claude asks for a credential — stop the session and report to Lead. Claude should never need your actual credentials to help you code.
Rotate Immediately Protocol
1. Stop the session immediately
2. Rotate the credential through the relevant service dashboard
3. Report to Lead — do not continue the session until confirmed
03
Mandatory Git Worktree for Every Task
REQUIRED
▼
Why Worktrees
  • Isolates Claude's file access to your card only
  • Prevents accidental modification of unrelated modules
  • Impossible to commit to the wrong branch
  • Clean, card-scoped directory Claude operates safely in
Branch Naming Convention
  • feature/FE-241-settings-plan-switcher
  • fix/BE-109-auth-refresh-token-expiry
  • hotfix/FE-255-safari-modal-z-index
Worktree Setup for Every Card
# Create worktree for your card
git worktree add ../salina-web-FE-241 feature/FE-241-settings-plan-switcher

# Navigate into it
cd ../salina-web-FE-241

# Start Claude Code here — scoped to this worktree only
claude

# After PR is merged — cleanup
git worktree remove ../salina-web-FE-241
git branch -d feature/FE-241-settings-plan-switcher
🚫
Never run Claude Code from the repo root or main/develop branch directly.

07
Dependency Install Review
REQUIRED
▼
Before approving any npm install or pip install:
  • Check the package name exactly — typosquatting is common (crossenv vs cross-env)
  • Verify the package is in your existing package.json or was explicitly discussed in the session
  • If Claude suggests a new dependency not in your card scope — deny it and check with Lead
  • After install, run npm audit and review the output before continuing
⚠️
Every PR triggers automated dependency scan. PRs with new dependencies flagged at high/critical severity are blocked from merge.

REQUIRED

◈ CLAUDE.md Security Block — Paste Into Every Project
## Security — Non-Negotiable

### Do Not Read or Reference
- .env, .env.local, .env.production, .env.staging
- Any file containing credentials, secrets, or API keys
- ~/.ssh/, ~/.aws/, ~/.config/
- Any file path outside the current worktree

### Do Not Suggest
- Hardcoded credentials, tokens, or passwords in code
- Reading sensitive paths for "context"
- Installing packages outside the card's defined scope
- Running commands as sudo or with elevated privileges
- Bypassing permission prompts

### All API Calls
- Must go through /lib/api/*.ts only
- Never raw fetch/axios to external URLs
- Never expose API keys in client-side code

### Credentials Are Referenced By Name Only
- process.env.DATABASE_URL — correct
- postgresql://user:secret@host — NEVER

◈ Core Concepts — What Developers Need to Know
🧩 Skills

Skills are reusable instruction sets that load into Claude's context at session start. Think of them as role-specific operating manuals that ensure Claude always knows your team's patterns, architecture, and standards.

  • Custom Skills — built by Lead, stored in your repo's .claude/skills/ folder
  • Partner Skills — Figma, Atlassian, Sentry, Hugging Face — enabled via MCP
  • Loading a Skill — referenced in your session prompt or auto-loaded via project config
  • Skill staleness — if the module changed, Skill must be rebuilt before use
# Reference a skill in your prompt
CONTEXT: [fe-best-practices] loaded

# Or attach the context MD directly
CONTEXT: docs/context/settings.context.md attached
🤖 Agents

In agentic mode, Claude executes multi-step tasks autonomously — reading files, running commands, making edits, running tests, then reporting back. You still approve each action via the permission system.

  • When to use agents — complex cards spanning multiple files or systems
  • When NOT to use — simple single-file changes (overkill, less predictable)
  • Supervision required — agent mode does not bypass the permission system
  • Subagents — Claude can spawn subagents for parallel tasks (Lead-only feature on Max)
⚠️
Agentic mode inherits your shell environment. Never run it with credentials exported in your shell.
🧠 Memory

Claude Code has no persistent memory between sessions by default. Memory is structured through files Claude reads at session start.

  • Project memory — CLAUDE.md in your project root. Read at every session start.
  • Global memory — ~/.claude/CLAUDE.md — applies to all projects. Good for personal preferences.
  • Context MDs — module-specific memory. Loaded manually or via Skills.
  • No cross-session memory — each session starts fresh. This is why context MDs matter.
# Global memory (applies everywhere)
~/.claude/CLAUDE.md

# Project memory (per repo)
[project-root]/CLAUDE.md

# Module memory (loaded per card)
docs/context/[module].context.md
📄 CLAUDE.md

The most important file in the entire workflow. Claude reads this at the start of every session in your project. It sets the rules, patterns, do-not-touch list, and security constraints that apply to every interaction.

# CLAUDE.md — Minimum Structure

## Project
[project name, purpose, tech stack]

## Architecture
[key module map — what files do what]

## Required Patterns
[coding standards, API call rules, state patterns]

## Do Not Touch
[files needing Lead approval before changes]

## Gotchas
[known bugs, Safari quirks, race conditions]

## Security — Non-Negotiable
[Do Not Read, Do Not Suggest — paste security block]
ℹ️
Lead provides a pre-filled CLAUDE.md template for each project at session start. Devs do not modify the Security section.

◈ The 8 Customizations

01
Set Up the cc Alias
Most Popular
▼

Type cc instead of the full claude command — skips permission prompts for faster workflow.

Prompt to Claude
Create an alias for me called cc that runs claude with --dangerously-skip-permissions
Added to ~/.zshrc or ~/.bashrc
alias cc='claude --dangerously-skip-permissions'
alias ccm='cc -c "make"'    # one-shot make
alias ccd='cc -c "dev"'     # one-shot dev
⚠️
Warning: --dangerously-skip-permissions allows edits and commands without asking. Only enable after months of safe usage and understanding the risks.

Linux note: cc may conflict with the C compiler — use any alias name you prefer.

02
Auto-Compact Before Context Degrades
Quality
▼

Prevents quality drop when the context window fills up. Default ~95% is too late — compact earlier.

Prompt to Claude
Set my auto-compact threshold to 75% in my user settings
~/.claude/settings.json
{
  "autoCompactThreshold": 0.75
}
Recommended Range
  • 60–75% for most people
  • 85% if you need more raw context
Related Commands
  • /clear — wipe history, start fresh
  • /compact — manual summarize
💡
Add to your CLAUDE.md: "During compaction, always preserve: file structure overview, current feature goal, key decisions, open questions."
03
Write Your Personal CLAUDE.md
Biggest Lever
▼

Applies globally to every session and project. Shapes Claude's style, tool choices, and behavior to match your workflow.

Prompt to Claude
Create my personal CLAUDE.md at ~/.claude/CLAUDE.md.
I prefer pnpm over npm, types over interfaces,
Vitest over Jest, and concise PR descriptions.
Example ~/.claude/CLAUDE.md
# Personal Claude Instructions

- Use pnpm, never npm or yarn
- Prefer TypeScript types over interfaces
- Testing: Vitest + React Testing Library
- PR descriptions: one-line summary + bullet points
- Keep answers concise unless asked for explanation
Rule Specificity Matters
  • Specific rules → ~89% compliance
  • Vague rules ("write clean code") → ~35%
  • Be concrete and actionable
Project-Level
  • Run /init in repo root
  • Heavily trim the generated result
  • Keep it focused on project-specific rules
04
Add a Live Status Line
Visibility
▼

Dashboard at the bottom of your terminal showing current directory, git branch, context usage %, and more.

Quick Setup
# Inside Claude Code, type:
/statusline

# This launches an interactive setup wizard
Custom Script (~/.claude/statusline-command.sh)
#!/bin/bash
# Shows cwd, branch, context % with color coding
# green = plenty of context, yellow = getting full, red = compact soon

BRANCH=$(git branch --show-current 2>/dev/null || echo "no-repo")
DIR=$(basename "$PWD")
echo "$DIR · $BRANCH"

Community options: ccstatusline, starship-claude integrations.

05
Auto-Format on Every Edit
Recommended
▼

Code stays formatted after every Claude edit — no manual Prettier/ESLint runs needed.

Prompt to Claude
Set up a PostToolUse hook in the project settings
that runs Prettier on any file after you edit or write it
Project .claude/settings.json
{
  "hooks": {
    "PostToolUse": [
      {
        "triggers": ["Edit", "Write"],
        "command": "pnpm prettier --write {file} || true"
      }
    ]
  }
}
💡
Extend it: Chain ESLint, tsc, Vitest, or any other tool in the hook pipeline.
06
Customize the Spinner Verbs
Fun
▼

Make waiting more fun. Replace the default loading text with custom messages.

Prompt to Claude
Replace my spinner verbs with these:
Hallucinating responsibly, Pretending to think,
Confidently guessing, Asking the void, Vibing with tokens
~/.claude/settings.json
{
  "spinnerVerbsOverride": [
    "Hallucinating responsibly",
    "Pretending to think",
    "Confidently guessing",
    "Asking the void",
    "Vibing with tokens"
  ]
}

Try themed sets: Harry Potter spells, Star Wars quotes, cooking verbs. Also customize spinnerTipsOverride.

07
Play a Sound When Claude Finishes
Productivity
▼

Multitask freely — get notified with an audio chime when a response is ready.

Prompt to Claude (macOS)
Set up a Stop hook in my user settings that plays
the Glass sound when you finish responding.
Use afplay on macOS.
~/.claude/settings.json
{
  "hooks": {
    "Stop": "afplay /System/Library/Sounds/Glass.aiff"
  }
}
macOS Sounds
  • Glass · Submarine · Purr
  • Funk · Pop · Frog · Hero
Cross-Platform
  • Linux: paplay sound.wav
  • Any .mp3/.wav with your player
