§ 0 Fill This In

§ Project Context

The first thing Claude reads. Without this, it guesses — and guessing means mistakes. Fill in every field once, per project.

Project Name The repo/app name. Claude uses this when writing commit messages, todo plans, and log entries. Don't leave as a placeholder.
Description 1–2 sentences on what the app does. Gives Claude enough business context to make sensible decisions without asking you.
Tech Stack List frameworks, libraries, and versions. e.g. Next.js 14, TypeScript, shadcn/ui, Tailwind, Neon PostgreSQL. Claude uses this to avoid suggesting incompatible patterns.
Key Directories Map your folder structure. Prevents Claude from creating files in the wrong places or asking where things live.
Environment Node version, how to run the dev server, any required env vars. Claude uses this for setup and debugging commands.
Deployment Where and how the app deploys. Affects how Claude writes build scripts, CI configs, and environment handling.
§ 1 Behavioral Rule

§ Plan Before Acting

Stops Claude from diving into code the moment it reads the task. Forces a thinking step first.

✎
Enter plan mode for non-trivial tasks
3+ steps or any architectural decision
▼
"Non-trivial" means anything with more than 3 steps, or any decision that affects architecture. A one-liner fix? Just do it. Adding a new feature? Plan first.
Plan gets written to tasks/todo.md as checkable items before touching code
User checks in on the plan before implementation starts — no silent autonomous runs
⊘
Stop and re-plan when things go sideways
Don't keep pushing through blockers
▼
Claude's biggest failure mode is compounding errors — hitting a wall, guessing a workaround, and making things worse. This rule creates a hard stop.
If mid-task execution fails, Claude stops, surfaces the blocker, and re-plans instead of improvising
Plan mode applies to verification steps too — not just build steps
§ 2 Behavioral Rule

§ Task Management Flow

A 6-step ritual that every task runs through — from first plan to final lesson. Click any step to learn more.

01
Plan → tasks/todo.md
Write checkable items to tasks/todo.md before touching any code. No plan = no execution.
writes: tasks/todo.md
02
Verify Plan
Check in with the user before starting implementation. The user reviews and approves the todo list — Claude doesn't self-authorize.
gate: user approval
03
Track Progress
Mark items complete with - [x] as work finishes. Gives the user a live view of progress without asking for updates.
updates: todo.md checkboxes
04
Explain Changes
At each step, give a high-level summary of what changed and why. Keeps the user informed without making them read diffs.
output: step summaries
05
Document Results
Add a review/result section to tasks/todo.md when the task is done. What worked, what was tricky, what changed from the plan.
writes: results to todo.md
06
Capture Lessons
After any user correction, log the pattern in tasks/lessons.md. This feeds into the Self-Improvement Loop in rule §3.
writes: tasks/lessons.md
§ 3 Behavioral Rule

§ Self-Improvement Loop

Claude accumulates corrections session by session. Over time, it makes fewer of the same mistakes.

✎
Log every correction to lessons.md
After any user correction, immediately
▼
Without this, Claude repeats the same mistakes across tasks. This rule creates a growing rulebook specific to your project and your corrections.
Entry is written as a preventive rule, not a log entry — "Don't do X, do Y instead"
File is reviewed at session start — Claude reads its own past mistakes before doing anything
If tasks/lessons.md doesn't exist, Claude creates it with header # Lessons Learned
§ 4 Behavioral Rule

§ Verification Before Done

A task isn't done because Claude says so. It's done when it provably works.

✓
Prove it works before marking done
Tests, logs, diffs — not just "I think it works"
▼
Claude has a tendency to say "done" optimistically. This rule forces a verification gate before the checkbox gets checked.
Run tests, check logs, demonstrate correctness — evidence required
Diff behavior between main and changes when the change is non-trivial
Internal check: "Would a staff engineer approve this?" — if unsure, don't mark done
§ 5 Behavioral Rule

§ Demand Elegance (Balanced)

Claude should challenge its own solutions — but only when it's worth the effort. The "balanced" qualifier is intentional.

◈
Pause and ask: "Is there a more elegant way?"
Non-trivial changes only — not simple fixes
▼
Prevents the two failure modes: hacky solutions that compound technical debt, and over-engineered solutions that waste time. The word "balanced" forces Claude to judge which situation it's in.
For non-trivial: if a solution feels hacky, step back and implement the clean version
For simple, obvious fixes: skip the elegance check entirely — don't over-engineer
Always challenge output before presenting it to the user
§ 6 Behavioral Rule

§ Autonomous Bug Fixing

When you hand Claude a bug, it handles it end-to-end. No back-and-forth, no hand-holding required.

↯
Given a bug report — just fix it
Zero context switching from the user
▼
Claude's default is to ask clarifying questions before acting on bugs. This rule disables that and makes Claude act like a senior engineer who owns the problem.
Point at logs, errors, and failing tests — then resolve them directly
Fix failing CI tests without being told how — explore the error, find the root cause
Only escalate back to the user if the fix requires a decision above Claude's scope
§ 7 Behavioral Rule

§ Subagent Usage

For complex problems, Claude can spin up sub-agents to run parallel workloads — keeping the main session clean and focused.

⊞
One focused task per subagent
Offload research, exploration, parallel analysis
▼
Context windows fill up. Subagents let Claude handle complex problems by distributing work — one agent per concern, results reported back to main.
Use subagents to keep the main context window clean — no polluting main with research noise
For complex problems: throw more compute via subagents, don't just try harder in a single context
Only active when Claude Code's agent mode is enabled — graceful fallback to single-agent
§ 8 Standards

§ Core Principles

Five non-negotiable behaviors that apply to every single task, regardless of context.

◇
Simplicity First
Make every change as simple as possible. Impact minimal code. The simplest solution that works is usually the right one.
◎
No Laziness
Find root causes. No temporary fixes. Senior developer standards apply — not "good enough for now."
✕
No Hallucination
If unsure, say so. Never invent APIs, file paths, or behaviors that haven't been confirmed to exist.
◆
Preserve Existing Code
Don't refactor what wasn't asked. Scope changes tightly. The work requested is the work done — nothing more.
△
No Orphaned Code
Remove code that is no longer used. No dead imports, no dead files, no unreachable exports left behind.
§ 9 Standards

§ Coding Conventions

Default conventions for the template. Customize these per project — Claude reads and follows them every session.

🎨 Formatting Prettier on save, ESLint strict mode. Claude won't introduce formatting inconsistencies.
🏷️ Naming camelCase for variables/functions — PascalCase for components — kebab-case for files
⚛️ Components Functional components only. No class components. Hooks over lifecycle methods.
📦 Imports Absolute paths via @/ alias. No relative ../../ climbing up the tree.
🔷 Types Prefer type over interface unless extending. Zero tolerance for any.
💬 Comments Only explain why, not what. Code should be self-documenting. No noisy inline comments.
§ 10 Hard Rules

§ Never Do

Absolute prohibitions. Claude treats these as zero-exception rules — no "just this once."

✕
Push to main directly. Always use a feature branch. Direct pushes to main bypass review and break team workflow.
✕
Use console.log in production code. Use the project logger. Stray logs pollute output and can leak sensitive data.
✕
Install packages without confirming. New dependencies have security, bundle size, and compatibility implications. Always ask first.
✕
Skip error handling. Every async operation needs a try/catch or .catch(). Unhandled rejections crash apps silently in production.
✕
Use any as a TypeScript escape hatch. It disables type safety and spreads. Find the right type, or use unknown with a type guard.
§ 11 File System

§ Tasks Directory

Two auto-managed files that Claude uses as its working memory within a project session.

project-root/tasks/
tasks/
├── todo.md     ← active task plan (Claude writes here)
└── lessons.md  ← corrections & patterns (Claude updates after mistakes)
✎
todo.md
Active task planning file
Claude writes the task plan here before touching code
Checkboxes tracked live as items are completed
Results section appended at end of task
Safe to delete between tasks — Claude recreates it
◉
lessons.md
Accumulated session memory
Grows over time — one entry per user correction
Claude reads it at session start — knows past mistakes
Entries written as prevention rules, not log entries
Add to .gitignore — session-scoped, not repo content
Bootstrap command: Run this once in your project root to initialize the tasks directory.
mkdir -p tasks && echo "# Active Task Plan" > tasks/todo.md && echo "# Lessons Learned" > tasks/lessons.md
