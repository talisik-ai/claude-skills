## Executive Summary

A lightweight internal web tool that lets MMI team members generate polished, professional resumes from a structured form. Input your details, preview in HTML, download as PDF. One template, zero friction.

## Problem Statement

### The Problem
Internal team members need professional resumes for client-facing profiles, proposals, and career development. Currently this is done manually in Word or Google Docs, leading to inconsistent formatting, wasted time, and outdated documents.

### Current State
Each team member maintains their own resume in various formats (Word, Google Docs, PDF). No standardized template. Updating requires manual formatting work. No central tool.

### Desired State
A web form where team members input their details once, preview a beautifully styled resume in real-time, and download a pixel-perfect PDF with one click. Consistent branding across all team resumes.

## Goals & Success Metrics

| Goal | Metric | Target |
|------|--------|--------|
| Reduce resume creation time | Time from start to finished PDF | < 10 minutes |
| Consistent formatting | All resumes use the same template | 100% adoption |
| Team adoption | Active users within first sprint | 80%+ of team |

## Scope

### In Scope (v1)
- Web-based form with structured input fields
- Real-time HTML preview of the resume
- Single professional template
- PDF download via browser print (CSS @media print)
- Form validation on required fields
- Plain HTML/CSS/JS implementation (no framework)

### Out of Scope
- Collaboration features (shared editing, comments)
- AI-generated content suggestions
- LinkedIn/external data import
- Multiple templates (future consideration)
- User authentication (internal tool, open access)
- Server-side PDF generation
- Data persistence / database storage

### Future Considerations
- Additional templates / theme switcher
- Save/load resume data (localStorage or backend)
- Export to DOCX format
- Team directory integration

## User Personas

### Internal Team Member
- **Role:** Developer, designer, PM, or lead at MMI
- **Goals:** Quickly generate a professional resume for proposals, clients, or career use
- **Pain Points:** Manual formatting is slow, resumes look inconsistent across the team, updating is tedious

## User Stories & Acceptance Criteria

### Story 1: Fill Resume Form
**As a** team member, **I want to** fill in my professional details in a structured form, **so that** I don't have to worry about layout or formatting.

**Acceptance Criteria:**
- [ ] Form includes fields: name, title, contact info, summary, experience (multiple), education (multiple), skills
- [ ] Required fields are validated before generation (name, title, at least one experience entry)
- [ ] User can add/remove multiple experience and education entries
- [ ] Form is responsive and usable on desktop

### Story 2: Preview Resume
**As a** team member, **I want to** see a live HTML preview of my resume, **so that** I can verify the content before downloading.

**Acceptance Criteria:**
- [ ] Preview updates as form fields are filled
- [ ] Preview matches the final PDF output
- [ ] Preview displays in a professional, clean layout

### Story 3: Download as PDF
**As a** team member, **I want to** download my resume as a PDF, **so that** I can share it externally.

**Acceptance Criteria:**
- [ ] PDF is generated via browser print / CSS @media print
- [ ] PDF output is single or multi-page with clean page breaks
- [ ] PDF matches the HTML preview layout exactly
- [ ] Download works in Chrome, Firefox, and Safari

## Technical Considerations

- **Stack:** Plain HTML, CSS, JavaScript — no framework dependencies
- **PDF generation:** Browser-native `window.print()` with `@media print` CSS
- **Hosting:** Can be served as a static file within any MMI project or standalone
- **No backend required** — all processing happens client-side
- **Template:** Single professional template with clean typography, proper spacing, and MMI-consistent styling

## Risks & Assumptions

| Type | Description | Mitigation |
|------|-------------|------------|
| Assumption | Browser print CSS produces consistent PDF across browsers | Test on Chrome, Firefox, Safari — Chrome is the baseline |
| Assumption | One template is sufficient for v1 | Gather feedback post-launch, add templates in v2 if needed |
| Assumption | No authentication needed for internal tool | Deploy on internal network or behind VPN |
| Risk | PDF page breaks may cut content awkwardly | Use CSS `break-inside: avoid` on sections |
| Risk | Low adoption if form is tedious to fill | Keep form minimal — only essential fields, smart defaults |
