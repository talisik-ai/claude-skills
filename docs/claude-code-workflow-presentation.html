Before the Pipeline

Step 0 — Project Bootstrap

The 8-step pipeline depends on three files existing. Nothing in the workflow runs correctly without them. Run this once per new project or legacy onboarding.

Step 0
Project Bootstrap
Owner: Lead
When: Once per new project / legacy onboarding
Tool: Claude Code (Max) + /project-bootstrap
/project-bootstrap
Three files must exist before Sprint 1
ARCHITECTURE.md Module map, stack overview, key boundaries, external integrations
docs/context/[module].context.md One per module — key files, dependencies, do-not-touch list, gotchas, patterns
CONTEXT-HEALTH.md Staleness tracker — module list, last verified sprint, owner, status per module
BOOTSTRAP Mode · New Project

Claude interviews the Lead in 3 structured rounds (9 questions) about modules, stack, team, and constraints. Generates all three files from the interview output.

ONBOARD Mode · Existing Codebase

Claude scans the codebase, infers stack and modules, then interviews the Lead to fill gaps — owners, do-not-touch lists, gotchas. Generates all three files without overwriting existing context MDs.

⚠

Gate: No Sprint 1 cards are created until the Lead has signed last_verified_sprint on every context MD. UNVERIFIED context MDs are not valid session sources.

The Core Workflow

The 8-Step Pipeline

Every feature, every MVP, every hotfix follows this sequence. Click each step to expand details. No step is skippable.

1
PRD Received
PM / Stakeholder Lead
The only valid trigger for development. No PRD = no card = no session.

A valid PRD must include

  • Clear objective
  • Scope — what's IN and what's explicitly OUT
  • At least one verifiable acceptance criterion
  • PRD number (e.g. PRD-042)
⚠

Gate: If the PRD is incomplete, Lead returns it. Work does not start on a partial PRD. No exceptions.

2
Impact Assessment
Lead
Claude reads ARCHITECTURE.md, the PRD, and all active context MDs to produce a full impact analysis.
/prd-assess docs/prd/PRD-042.md

Output: IMPACT-ASSESSMENT.md contains

  • PRD type — New Feature / New Project / Pivot / Hotfix / Cross-Project
  • Affected modules and their role owners
  • Context MDs that need updating before work begins
  • Custom Skills that need rebuilding
  • Estimated Jira cards per role
  • Dependency and cross-project risks
⚠

Gate: Nothing moves forward until the IMPACT-ASSESSMENT.md exists and the Lead has reviewed it.

3
Context MDs Updated
Lead
Every module listed in the Impact Assessment gets its context MD updated and signed.

Per affected module

  • Update context MD — key files, dependencies, do-not-touch list, gotchas, required patterns
  • Update last_verified_sprint to current sprint (Lead sign-off)
  • Update ARCHITECTURE.md if module structure changed
  • Rebuild the Custom Skill for that module
  • Update CONTEXT-HEALTH.md — staleness flags, ownership changes
⚠

Warning: A stale context MD is worse than none. Developers working from stale context build confidently in the wrong direction.

4
Jira Cards Created
Lead
Claude generates the full card list from the impact assessment. Lead reviews before creating in Jira.
/gen-cards docs/context/IMPACT-ASSESSMENT.md

Every card must have

  • PRD reference — mandatory, no card without this
  • Skill name or context MD path
  • Exact scope: files the developer MAY touch
  • Explicit do-not-touch list
  • Acceptance criteria (verifiable)
  • Size: XS, S, or M only — no L or XL cards
  • 3 pre-set log fields: AI-assisted (Y/N) · rework required (Y/N) · session count
⚠

Rule: Cards larger than M are split before creation. A card that's too large will cause the developer to hit session limits mid-implementation.

5
Implementation
Developer
One session per card. Claude Pro + role Skill + project context Skill.

Session start sequence

  • Open Claude Pro — confirm role Skill is enabled (fe-best-practices / be-best-practices / etc.)
  • Check if a project Skill exists for the module — if yes, it auto-loads. If no, attach the context MD manually
  • Open session using the mandatory prompt template

Mandatory session prompt template

PRD: PRD-042 — [PRD title] TASK: [exact task from Jira card] SCOPE: Touch ONLY [files listed on card] DO NOT: [do-not-touch list from card] CONTEXT: [Skill name] loaded / [module.context.md] attached DONE WHEN: [acceptance criteria]

Session rules

  • One session per card — close and start fresh for each card
  • When acceptance criteria are met, stop — do not keep going
  • Limit hit mid-session = card was scoped too large → flag and split, do not continue
  • Never paste raw source files into Claude — use Skills or attach context MDs only
  • On card close, developer fills the 3 log fields before moving to Done
6
Review Buddy + QA
Review Buddy QA
Two-gate system. Buddy runs first, QA runs after Buddy sign-off only.

Review Buddy checklist (pre-review Skill)

  • Did the PR only touch files listed on the card?
  • Is acceptance criteria verifiable from the code?
  • No untouched-file modifications
  • No schema changes outside scope
  • Patterns match the coding-standards Skill

QA (after Buddy sign-off)

  • Generate test cases from acceptance criteria
  • Generate edge cases — empty states, boundary values, error paths
  • Browser matrix — Chrome, Firefox, Safari
  • Regression check on files adjacent to scope boundary
⚠

Rule: Review Buddy and QA must be different people. One person doing both defeats the two-gate system. Bug found by QA → returns to Step 5.

7
Lead PR Review
Lead
Target: less than 30% of all PRs. Lead reviews only what requires architectural judgment.

Lead reviews only

  • PRs touching ARCHITECTURE.md, /lib/auth/, or cross-module dependencies
  • New patterns not in the coding-standards Skill
  • Pivot-adjacent code flagged high-risk in the Impact Assessment
  • Anything the Buddy escalates

After merge — Lead closes the loop

  • New pattern established → update coding-standards Skill immediately
  • Module structure changed → update context MD, re-sign last_verified_sprint, rebuild Skill
  • Module ownership changed → update CONTEXT-HEALTH.md
  • Version bump ARCHITECTURE.md if structural changes shipped
⚠

Warning: If a Lead is reviewing every PR, they are not doing Impact Assessments. Impact Assessments are the higher-value use of Lead capacity.

8
Deploy + Docs
DevOps Lead
CI/CD deploy, 30-min Sentry watch, then Lead closes the docs loop.

DevOps

  • Deploy via CI/CD to self-hosted infrastructure
  • Smoke test on staging before promoting to production
  • Monitor Sentry for 30 min post-deploy
  • Log deployment against PRD number in devops-infra Skill changelog

Lead post-deploy

  • Version bump ARCHITECTURE.md
  • Confirm last_verified_sprint is current on all touched context MDs
  • Rebuild affected Skills
  • Update CONTEXT-HEALTH.md — all touched modules marked fresh
  • Run monthly metric review if end-of-sprint

Non-Negotiable

11 Rules

The workflow only works if the rules are followed. Every rule exists because a gap in it causes compounding failure downstream.

01
PRD first, Jira second.No card without a completed Impact Assessment. Full stop.
02
Skills support the workflow — they don't define it.The PRD → Impact Assessment → Context MD → Card → Session pipeline is immovable.
03
No raw codebase dumping.Context MDs or Skills only. Never paste raw source files into Claude.
04
One session per card.Close and restart for each card. No thread kept alive across multiple cards.
05
Pivot = disable old Skills immediately.Notify all developers before Skills are re-enabled. Stale Skills on a pivot destroy sprint work.
06
Leads handle "why", devs handle "how".Architectural decisions stay with Leads on Max tier. Devs do not make architectural calls in sessions.
07
Prompt template is mandatory.PRD ref, task, scope, do-not-touch, done criteria — in every single session. No exceptions.
08
Limit hits are planning signals.Card was too large. Split it. Track limit events as a sprint health metric — they indicate scoping failure.
09
Measure every sprint.AI-assist rate, 1-session completion, rework rate. Without measurement, there is no improvement and no accountability.
10
Review Buddies protect Lead capacity.Lead reviews <30% of PRs. If Lead reviews everything, Impact Assessments don't get done.
11
Stale context MDs are worse than none.Every context MD carries a last_verified_sprint. If it's not signed off this sprint, it is not a valid source. Period.
The Reality Check

The Team Filter

This workflow makes the gap between developers visible. The measurement system makes it objective, not subjective. After 2 sprints, the data tells the story.

✓

Developer Using the Workflow

→Starts every session from a clean, scoped context
→Completes most cards in one session
→Fills log fields — contributing to sprint measurement data
→PRs pass Review Buddy on first submission
→Asks Claude the implementation questions, not the Lead
Multiplier. Gets more done with less effort every sprint.
✕

Developer Not Using the Workflow

→Starts sessions without loading context → asks Lead for direction
→Needs multiple sessions per card → hits limits → blames card scope
→Skips log fields → sprint measurement is incomplete
→PRs fail Buddy review → rework → delay for everyone
→Is the bottleneck the workflow was designed to eliminate
The measurement system (Rule 09) makes this objective. After 2 sprints, the data decides.
